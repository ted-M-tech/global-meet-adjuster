# UI/UX Design Review & Improvement Proposals
## Global Meet Adjuster

---

## 1. User Flow Analysis (Host / Guest)

### Current Spec Assessment
仕様書のユーザーフローは全体として論理的だが、以下のUX改善ポイントがある。

### Host Flow Improvements

**Issue 1: ログイン〜イベント作成のステップ数**
現在の仕様: `/` (ログイン) → `/dashboard` → `/events/new` → 共有URL発行
提案: 初回ログイン時は「最初のイベントを作成しましょう」というオンボーディングで直接 `/events/new` に誘導する。ダッシュボードが空の状態（Empty State）で迷子にさせない。

**Issue 2: 共有URL発行後の導線が不明確**
イベント作成後に「次に何をすべきか」が仕様に不足している。
提案:
```
┌─────────────────────────────────────────────┐
│  ✅ イベントが作成されました！               │
│                                              │
│  共有URL:                                    │
│  ┌────────────────────────────┬──────────┐  │
│  │ https://example.com/ev/abc │ コピー📋 │  │
│  └────────────────────────────┴──────────┘  │
│                                              │
│  [LINEで送る] [メールで送る] [QRコード表示]  │
│                                              │
│  ───────────────────────────────────────     │
│  💡 参加者がURLにアクセスして回答すると、     │
│     ここに回答状況が表示されます。            │
│  [イベント詳細を見る →]                      │
└─────────────────────────────────────────────┘
```

**Issue 3: OAuth再認証フローのUX**
日程確定時にOAuthトークン失効していると再認証が必要。これはユーザーにとって唐突。
提案:
- 「決定」ボタン押下前に、トークン状態を事前チェック
- 失効時は「Google Calendarに登録するには再度ログインが必要です」と明示し、インラインで再認証ボタンを表示
- 再認証完了後、自動的に確定処理を続行（ユーザーが再度「決定」を押す必要がないように）

### Guest Flow Improvements

**Issue 4: ゲストが最初に見る画面の情報設計**
ゲストは共有URLを開いた瞬間に「何をすべきか」を即座に理解する必要がある。
提案: ヒーロー領域を設けて、コンテキストを明示する。
```
┌─────────────────────────────────────────────┐
│  📅 チームミーティング日程調整               │
│  主催: 田中太郎                              │
│  所要時間: 1時間                             │
│  メモ: 月例のチーム定例です                  │
│                                              │
│  ┌─────────────────────────────────────┐    │
│  │  あなたの都合を回答してください      │    │
│  │  [新しく回答する]                    │    │
│  └─────────────────────────────────────┘    │
│                                              │
│  現在の回答状況（3人回答済み）               │
│  ┌─────────────────────────────────────┐    │
│  │  [投票テーブル]                      │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────┘
```

**Issue 5: ゲスト名前・メール入力のタイミング**
現仕様ではDialogで名前・メールを入力してから投票する。しかし、ゲストは「まず候補日を見たい」→「回答したい」→「情報入力」の順が自然。
提案:
- 先に投票テーブルを表示し、回答入力を可能にする
- 「保存する」押下時に名前・メールをBottomSheetまたはインラインで入力
- これにより、ゲストの離脱率を下げられる（先に情報を求めると離脱する心理的障壁がある）

## 2. Responsive Design Strategy

### PC Layout (≥1024px)
横長テーブル形式。調整さんと同等のレイアウト。
```
┌──────────────────────────────────────────────────────────────┐
│           │ 1/15(月)  │ 1/16(火)  │ 1/17(水)  │ 1/18(木)  │
│           │ 19:00〜   │ 19:00〜   │ 20:00〜   │ 19:00〜   │
├──────────────────────────────────────────────────────────────┤
│ 田中太郎  │    ◯      │    △      │    ◯      │    ×      │
│ 鈴木花子  │    ◯      │    ◯      │    ×      │    ◯      │
│ 佐藤一郎  │    △      │    ◯      │    ◯      │    ◯      │
├──────────────────────────────────────────────────────────────┤
│ ◯ 合計   │    2      │    2      │    2      │    2      │
│ △ 合計   │    1      │    1      │    0      │    0      │
│ × 合計   │    0      │    0      │    1      │    1      │
└──────────────────────────────────────────────────────────────┘
```
- 列ヘッダーは sticky（横スクロール時に名前列を固定）
- ◯の数が最大の列をハイライト表示（薄い緑背景）
- ホスト画面では各列に「この日程で決定」ボタンを表示

### Tablet Layout (768px〜1023px)
テーブルレイアウトを維持しつつ、列幅をコンパクトに。
- 日付表示を `1/15(月)` のように短縮
- 名前列幅を縮小
- 横スクロール可能にし、名前列は sticky position で固定

### Mobile Layout (<768px)
**カード形式に完全切り替え** — これが最重要のレスポンシブ対応。

```
┌─────────────────────────────────┐
│  📅 1月15日（月）19:00〜20:00   │
│  あなたの時間: 1/16 11:00 JST  │
│                                 │
│  ┌─────┐ ┌─────┐ ┌─────┐      │
│  │  ◯  │ │  △  │ │  ×  │      │
│  │ OK  │ │微妙 │ │ NG  │      │
│  └─────┘ └─────┘ └─────┘      │
│                                 │
│  回答状況:                      │
│  田中: ◯  鈴木: ◯  佐藤: △    │
└─────────────────────────────────┘

┌─────────────────────────────────┐
│  📅 1月16日（火）19:00〜20:00   │
│  あなたの時間: 1/17 11:00 JST  │
│  ...                            │
└─────────────────────────────────┘
```

**カード形式の利点:**
- 各候補日が独立したカードとなり、縦スクロールで自然に操作できる
- ◯△×ボタンが十分な大きさ（最小タップターゲット: 44x44px）
- When2meetの「PCレイアウトをそのままモバイルに表示」する問題を回避

**切り替えの実装方針:**
- Tailwind CSSの `md:` ブレークポイントで分岐
- テーブルコンポーネントとカードコンポーネントを別々に実装し、条件レンダリング
- データソースは共通（表示コンポーネントだけ切り替え）

```
│                                 │
```

## 3. Timezone Display UX

### 現仕様の課題
仕様書は「ゲストのブラウザのタイムゾーンに自動変換」「ホストのタイムゾーンも併記」としているが、具体的な表示方法が未定義。

### 提案: Dual Timezone Display

#### PC表示
```
┌──────────────────────────────────┐
│ 1月15日（月）19:00〜20:00        │
│ 🌏 PST (ホスト時間)             │
│ ─────────────────────           │
│ あなたの時間:                    │
│ 1月16日（火）12:00〜13:00 JST   │
└──────────────────────────────────┘
```

#### 表示ルール
1. **プライマリ**: ゲストのローカル時間を大きく表示（これがゲストにとって最も重要）
2. **セカンダリ**: ホスト時間を小さく併記（参考情報）
3. **日付ずれの明示**: タイムゾーン変換で日付が変わる場合、日付も明示する（例: PST 1/15 → JST 1/16）
4. **タイムゾーン名の表示**: 「JST」「PST」だけでなく、「Asia/Tokyo (JST)」のようにIANA名も表示可能にする

#### タイムゾーン切り替えUI
```
┌────────────────────────────────────────┐
│ 表示タイムゾーン:                       │
│ [● 自動検出 (Asia/Tokyo)] [○ ホスト時間 (America/Vancouver)] │
│ [○ カスタム ▼]                         │
└────────────────────────────────────────┘
```
- デフォルトはブラウザのタイムゾーンで自動検出
- ワンクリックでホスト時間に切り替え可能
- VPN等でタイムゾーンが正しく検出されない場合のためにカスタム選択も用意

#### 日付跨ぎの視覚的警告
タイムゾーン変換で日付が変わる場合、特に注意喚起が必要:
```
┌─────────────────────────────────────────┐
│  ⚠️ ホストの1月15日(月)は、              │
│  あなたの時間では1月16日(火)です         │
│                                          │
│  📅 1月16日（火）12:00〜13:00 JST       │
│  (ホスト: 1/15 19:00 PST)               │
└─────────────────────────────────────────┘
```

## 4. AI Date Assist UX Flow

### 現仕様の課題
「テキストエリアに自然言語を入力 → Gemini APIが解析 → 候補日提案」は基本フローとして良いが、具体的なインタラクションが未定義。

### 提案: 3-Step AI Assist Flow

#### Step 1: Natural Language Input
```
┌─────────────────────────────────────────────┐
│  📝 候補日をAIで設定                        │
│                                              │
│  ┌─────────────────────────────────────┐    │
│  │ 来週の月水金 19:00から              │    │
│  │                                      │    │
│  └─────────────────────────────────────┘    │
│                                              │
│  💡 入力例:                                  │
│  ・「来週の平日 19:00」                      │
│  ・「1月20日〜24日の午後」                   │
│  ・「今月の水曜と金曜 10:00」               │
│  ・「再来週の月火水 18:30から」              │
│                                              │
│  [AIで候補日を生成 ✨]                       │
└─────────────────────────────────────────────┘
```

**UXポイント:**
- プレースホルダーテキストで入力フォーマットのヒントを表示
- テキストエリアの下に具体的な入力例をチップ形式で表示（クリックで入力される）
- Enterキーで送信 or ボタンクリック

#### Step 2: AI Processing & Result
```
┌─────────────────────────────────────────────┐
│  ⏳ AIが候補日を生成中...                    │
│  ┌─────────────────────────────────────┐    │
│  │ ████████████░░░░░░░░░  60%          │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────┘

        ↓ (1〜3秒後)

┌─────────────────────────────────────────────┐
│  ✅ 3件の候補日が生成されました              │
│                                              │
│  「来週の月水金 19:00から」を解析した結果:   │
│                                              │
│  ☑️ 1月20日（月）19:00〜20:00               │
│  ☑️ 1月22日（水）19:00〜20:00               │
│  ☑️ 1月24日（金）19:00〜20:00               │
│                                              │
│  [✏️ 修正する] [➕ 候補日を追加]             │
│                                              │
│  [この候補日で確定 →]                        │
└─────────────────────────────────────────────┘
```

**UXポイント:**
- Skeleton UIでローディング表示（スピナーよりもコンテンツ予測が立つ）
- 解析結果を確認しやすいチェックボックスリストで表示
- 不要な候補日はチェックを外して除外可能
- エラー時:「入力を理解できませんでした。もう少し具体的に入力してください」+ リトライ可能

#### Step 3: Edit & Finalize
```
┌─────────────────────────────────────────────┐
│  候補日一覧  (ドラッグで並べ替え可能)        │
│                                              │
│  ≡ 📅 1/20（月）19:00〜20:00  [✏️] [🗑️]    │
│  ≡ 📅 1/22（水）19:00〜20:00  [✏️] [🗑️]    │
│  ≡ 📅 1/24（金）19:00〜20:00  [✏️] [🗑️]    │
│                                              │
│  ───────────────────────────────────────     │
│  ➕ 手動で候補日を追加                       │
│  ┌──────────────┬──────────┐                │
│  │ 日付を選択 📅 │ 時間 🕐  │                │
│  └──────────────┴──────────┘                │
│  [追加]                                      │
│                                              │
│  ✨ AIで追加の候補日を生成                   │
│                                              │
│  ───────────────────────────────────────     │
│  [イベントを作成 →]                          │
└─────────────────────────────────────────────┘
```

**UXポイント:**
- AI生成と手動追加を統合した候補日管理画面
- 各候補日にインライン編集ボタン（日時を個別修正可能）
- ドラッグ&ドロップで順序変更（表示順 = 投票テーブルの列順）
- AIで「追加の候補日を生成」でき、既存候補と重複しないよう配慮

## 5. Voting UI (◯△×) Interaction Design

### タップ/クリックのインタラクション

#### 方式: トグルサイクル方式
各セルをクリック/タップするたびに状態が循環: `未回答 → ◯ → △ → × → ◯ → ...`

**利点:**
- ワンタップで操作完了（ラジオボタン方式より速い）
- 調整さんユーザーにとって馴染みのある操作
- テーブル形式で多数の候補日を素早く回答できる

#### PC: テーブルセルクリック
```
┌────────────────────────┐
│         1/15(月)       │
│         19:00〜        │
├────────────────────────┤
│                        │
│     ◯                  │  ← クリックで ◯→△→×→◯ と循環
│     OK                 │
│                        │
└────────────────────────┘
```

#### Mobile: カード内ボタン（3ボタン方式）
モバイルではトグルサイクルではなく、3つの明示的なボタンを使う。
```
┌──────────────────────────────┐
│   ┌────┐  ┌────┐  ┌────┐   │
│   │ ◯ │  │ △ │  │ × │   │
│   │ OK │  │微妙│  │ NG │   │
│   └────┘  └────┘  └────┘   │
└──────────────────────────────┘
```

**理由:** モバイルでは「現在の状態」と「タップしたら何になるか」が分かりにくいため、3ボタンで明示的に選択させる方が誤操作を防げる。

### 視覚デザイン

| 状態 | 色 | アイコン | 背景色 |
|------|-----|---------|--------|
| ◯ (OK) | Green (#22C55E) | CircleCheck | bg-green-50 |
| △ (Maybe) | Amber (#F59E0B) | Triangle | bg-amber-50 |
| × (NG) | Red (#EF4444) | XCircle | bg-red-50 |
| 未回答 | Gray (#9CA3AF) | Minus | bg-gray-50 |

### キーボード操作
- Tab: 次のセルに移動
- 1 / o: ◯を設定
- 2 / m: △を設定
- 3 / x: ×を設定
- Space / Enter: 次の状態に循環
- 矢印キー: テーブル内の移動

## 6. Accessibility (WCAG 2.1 AA)

### Color & Contrast
- すべてのテキストはWCAG AA基準のコントラスト比（4.5:1以上）を満たす
- ◯△×は色だけでなくアイコン形状でも区別可能（色覚多様性対応）
- フォーカスリングは `ring-2 ring-offset-2 ring-blue-500` で視認性を確保

### Keyboard Navigation
- 全機能がキーボードのみで操作可能
- 投票テーブルは `role="grid"` + 矢印キーナビゲーション
- フォーカストラップ: Dialog表示中はDialog内にフォーカスを閉じ込める
- Skip Navigation: ページ先頭に「メインコンテンツへスキップ」リンクを配置

### Screen Reader Support
- 投票テーブルの各セルに適切な `aria-label` を付与
  - 例: `aria-label="1月15日 19時の候補に対する回答: OK"`
- 状態変更時に `aria-live="polite"` でアナウンス
  - 例: 「回答を◯に変更しました」
- テーブルの集計行にも適切なラベルを付与
- ローディング状態: `aria-busy="true"` + `role="status"` で通知

### Motion & Animation
- `prefers-reduced-motion` メディアクエリでアニメーションを無効化可能
- 自動再生するアニメーションは使用しない

### Form Accessibility
- すべてのフォーム要素に `label` 要素を関連付け
- エラーメッセージは `aria-describedby` でフォーム要素に関連付け
- 必須フィールドに `aria-required="true"` を設定

## 7. Error / Empty / Loading States

### Loading States

#### Skeleton UI Pattern
```
┌─────────────────────────────────────────┐
│  ████████████████  ← タイトル            │
│  ████████  ████████████                  │
│                                          │
│  ┌──────┬──────┬──────┬──────┐          │
│  │ ████ │ ████ │ ████ │ ████ │  ← テーブル│
│  │ ████ │ ████ │ ████ │ ████ │          │
│  │ ████ │ ████ │ ████ │ ████ │          │
│  └──────┴──────┴──────┴──────┘          │
└─────────────────────────────────────────┘
```
- スピナーではなくSkeleton UIを採用（コンテンツの配置を予測できる）
- AI候補日生成中はパルスアニメーション付きのSkeleton
- ボタンのローディング中は `disabled` + スピナーアイコン

#### インライン処理表示
```
[Google Calendarに登録中... ⏳]  →  [✅ 登録完了！]
```
- ページ遷移なしでインラインで処理状態を表示
- Toast通知で成功/失敗を伝える

### Empty States

#### ダッシュボード（イベントなし）

```
│                                          │
│    [最初のイベントを作成 →]              │
│                                          │
```

#### 投票画面（回答者なし）

```
┌─────────────────────────────────────────┐

│    最初の回答者になりましょう！           │
│    [回答する →]                          │
│                                          │
└─────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────┐
```

### Toast Notifications
- **Success**: 緑（保存完了、回答完了）
- **Error**: 赤（通信エラー、保存失敗）
- **Info**: 青（コピー完了、タイムゾーン変更通知）
- **Warning**: 黄（トークン失効、回答期限間近）
- 位置: 画面右下（PC）、画面下部中央（モバイル）
- 自動非表示: 5秒後（エラーは手動で閉じるまで表示）

### Global Meet Adjuster の差別化ポイント

1. **AI候補日入力**: 自然言語で「来週の月水金19時」と入力するだけで候補日生成。競合には無い機能
2. **自動タイムゾーン変換**: 調整さんにはない機能。When2meetはあるがモバイルで使いにくい
3. **モバイルファーストの投票UI**: カード形式で快適なモバイル体験。When2meetとの最大の差別化
4. **Google Calendar + Meet 一体連携**: 日程確定→カレンダー登録→Meet URL発行がワンクリック
5. **現代的なデザイン**: shadcn/ui + Tailwind CSSによる洗練されたUI。調整さんとの視覚的差別化

## 10. Micro-interactions & Animations

### 投票インタラクション
- **◯△×切り替え**: `scale(0.95)` → `scale(1)` の短いバウンスアニメーション (150ms, ease-out)
- **色遷移**: 背景色の変化は `transition-colors duration-200` でスムーズに
- **Haptic Feedback**: モバイルでは投票時に微弱なバイブレーション（Vibration API）

### カード/リスト
- **候補日カードの追加**: `fadeIn` + `slideUp` (300ms) でリストに追加される
- **候補日カードの削除**: `fadeOut` + `slideLeft` (200ms) で消える
- **ドラッグ&ドロップ**: 掴んだカードに `shadow-lg` + `scale(1.02)` を適用
- **カードの並べ替え**: 他のカードが `transform` でスムーズに移動

### ページ遷移
- **ページ間遷移**: `framer-motion` の `AnimatePresence` で `fadeIn` (200ms)
- **Dialog表示**: `scale(0.95, 1)` + `opacity(0, 1)` (200ms) — shadcn/uiデフォルト活用

### フィードバック
- **コピーボタン**: クリック後に一時的にチェックマークアイコンに切り替わり、1.5秒後に元に戻る
- **保存ボタン**: 処理中はスピナー表示、完了後にチェックマークに変わり、1秒後にテキストに戻る
- **Toast**: 右下からスライドイン (300ms)、5秒後にフェードアウト

### スクロール連動
- **テーブルヘッダー**: スクロール時に `sticky` + `shadow-sm` が追加される（浮いている感を演出）
- **名前列固定**: 横スクロール時に名前列の右端にシャドウ効果

### ローディング
- **Skeleton Pulse**: `animate-pulse` (Tailwind標準) でコンテンツ読み込み中を表現
- **AI生成中**: タイプライター風のドットアニメーション「AI が解析中...」
- **プログレスバー**: 候補日数に応じた段階的な進捗表示
